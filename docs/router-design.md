# 智能模型路由技术方案

## 一、背景与痛点

### 1.1 业务场景

GopherAI 平台支持多种后端模型：OpenAI 主模型（复杂推理）、RAG 知识库（文档问答）、MCP 工具调用（外部系统操作）、Ollama 本地模型（低成本闲聊）。前端通过 `modelType=auto` 委托后端自动选择最优模型，核心诉求是 **在不牺牲回答质量的前提下最小化推理成本**。

### 1.2 现状

当前使用 `RuleBasedRouter`（关键词 + 字符长度启发式规则），路由决策优先级如下：

```
MCP 关键词命中 → MCP 工具调用
RAG 关键词命中  → RAG 知识检索
复杂关键词/长文本 → OpenAI 主模型
问候关键词/极短文本 → Ollama/低成本模型
以上均未命中 → 兜底主模型
```

### 1.3 核心痛点

| 痛点 | 示例 | 影响 |
|------|------|------|
| **语义盲区** | "帮我看看这个怎么回事"（用户已上传文档）→ 关键词不命中 → 走主模型而非 RAG | 回答质量下降，成本浪费 |
| **关键词歧义** | "简单分析一下这段代码" → 命中"简单" → 走廉价模型 → 输出质量差 | 用户需手动切换模型 |
| **无法改写** | 口语化/指代模糊 query 直接透传（"上次那个价格怎么算的来着"）→ RAG 检索效果差 | 知识库利用率低 |
| **维护成本** | 新增意图类型需改代码 + 补充关键词列表 + 发布 | 迭代效率低 |
| **排他路由** | 每个 query 只能选一种模型，不支持"先检索文档再让主模型推理"的混合模式 | 能力上限受限 |

---

## 二、需求定义

### 2.1 功能需求

- **FR1**: 根据用户 query 的语义内容自动选择最合适的后端模型
- **FR2**: 当 query 模糊/口语化时，能改写为规范表达后再路由
- **FR3**: 支持降级兜底，任何路由组件失效不影响服务可用
- **FR4**: 路由决策需可观测（命中规则、置信度、耗时）
- **FR5**: 前端不感知路由逻辑，统一使用 `modelType=auto`

### 2.2 非功能需求

| 维度 | 目标 |
|------|------|
| 分类延迟增量 | <300ms（理想 <100ms） |
| 额外成本增量 | 趋近于零（优先复用已有基础设施） |
| 分类准确率 | >85%（相对人工标注） |
| 可用性 | 99.9%，分类组件故障自动降级 |

## 三、候选方案

### 3.1 方案总览

```
                    分类方式
              静态           动态
          ┌──────────┬──────────────┐
    直接  │ ① 关键词  │ ② Embedding   │

          └──────────┴──────────────┘
```

### 3.2 方案一：规则路由器（当前基线）

**原理**：维护关键词集合，对 query 做大小写不敏感的 `strings.Contains` 匹配 + 字符长度阈值判断。

**路由决策链**：
```
MCP 关键词命中 → MCP
RAG 关键词命中  → RAG
复杂关键词/长度≥300 → 主模型
问候关键词/长度≤12 → 低成本模型
兜底 → 主模型
```

| 维度 | 评价 |
|------|------|
| 延迟 | ★★★★★ 0ms |
| 成本 | ★★★★★ 零额外开销 |
| 准确率 | ★★☆☆☆ ~65%（语义盲区大） |
| 维护性 | ★★☆☆☆ 新增意图需改代码 |
| Query改写 | ✗ 不支持 |
| 混合意图 | ✗ 不支持 |
| 冷启动 | ★★★★★ 零 |

**适用**：MVP 阶段快速验证，或作为其他方案的降级兜底。

---

### 3.3 方案二：Embedding 语义匹配

**原理**：将路由建模为向量检索问题——为每类意图预置锚点样例 query 做 Embedding，在线计算用户 query 与各意图锚点质心的余弦相似度，取最相似意图。

```
离线：意图锚点样例 → Embedding → 向量库
在线：用户 query → Embedding → Top-K 相似意图 → 路由
```

| 维度 | 评价 |
|------|------|
| 延迟 | ★★★★★ ~5ms（纯数学运算） |
| 成本 | ★★★★★ 复用已有 Embedding 基础设施 |
| 准确率 | ★★★☆☆ ~80%（依赖锚点质量） |
| 维护性 | ★★★☆☆ 需持续维护锚点样例 |
| Query改写 | ✗ 不支持 |
| 混合意图 | ✗ 不支持 |
| 冷启动 | ★★☆☆☆ 需人工准备高质量锚点 |

**优点**：极快、极省，可复用已有 RAG 向量化设施。

**缺点**：泛化能力取决于锚点覆盖面；对多语言混合/口语化 query 容忍度低。

**适用**：意图类型稳定、query 模式可预测的成熟期产品，常作为 LLM 分类方案的前置快速层。

---

### 3.4 方案三：成本级联（Cascading）

**原理**：不预判意图，按模型成本从低到高逐级调用，每级判断回答质量是否达标，达标即返回，不达标升级到下一级。

```
L1: Ollama（免费）→ 质量检查 → 达标则返回
     ↓ 不达标
L2: RAG 增强（中成本）→ 质量检查 → 达标则返回
     ↓ 不达标
L3: 主模型（高成本）→ 兜底返回
```

**质量检查器**：
- 回答长度 ≥ 10 字符
- 不含"我不知道/无法回答/抱歉"等拒答模式
- 回答与 query 的 Embedding 余弦相似度 ≥ 0.3

| 维度 | 评价 |
|------|------|
| 延迟 | ★★☆☆☆ 最坏 L1+L2+L3 串行 |
| 成本 | ★★★★☆ 简单问题零成本 |
| 准确率 | ★★★★☆ 基于实际产出判断 |
| 维护性 | ★★★☆☆ 质量规则需持续调优 |
| Query改写 | ✗ 不支持 |
| 混合意图 | ★★★☆☆ 自然支持（级联过程隐含混合） |
| 冷启动 | ★★★★★ 零 |

**优点**：真正"能省则省"，决策基于实际回答质量而非预测。

**缺点**：最坏情况延迟叠加（L1+L2+L3）；流式场景不友好（每级都要等完整响应才能判断）。

**适用**：异步/批处理场景，对延迟不敏感、成本优先。

---

### 3.5 方案四：Bandit 在线学习

**原理**：将路由建模为 Contextual Multi-Armed Bandit 问题，从运行数据中自动学习 query 特征到模型选择的最优映射。

```
状态：query 特征向量（Embedding + 长度 + 时间 + 用户画像）
动作：选择哪个模型
奖励：-cost + α × quality_score

运行越久，策略越准。
```

| 维度 | 评价 |
|------|------|
| 延迟 | ★★★★★ 推理零延迟（查表/轻量模型） |
| 成本 | ★★★★☆ 收敛后更省 |
| 准确率 | ★★★★☆ 随时间提升 |
| 维护性 | ★★☆☆☆ 需特征平台 + 反馈采集 |
| Query改写 | ✗ 不支持 |
| 混合意图 | ★★★☆☆ 可建模多标签 |
| 冷启动 | ★☆☆☆☆ 需大量运行数据，启动期需规则兜底 |

**优点**：自进化，能发现人类设计之外的模式；多目标（成本/质量/延迟）可灵活权衡。

**缺点**：冷启动困难，需反馈信号（点赞/追问/停留时长）采集基础设施；环境变化（切换模型）历史数据可能失效。

**适用**：有流量的成熟产品，已有用户行为埋点体系。

---

### 3.6 方案五：轻量 LLM 语义分类（当前实现）

**原理**：在路由前插入一次极轻量的 LLM 调用，用自然语言 Prompt 做意图分类 + Query 改写，输出结构化 JSON 决策。

```
用户 query
    ├─ [短路] 高频问候词 → 直达低成本模型（跳过 LLM）
    └─ [分类] 轻量 LLM (Ollama 优先，OpenAI 降级)
           ├─ 输出: {intent, confidence, rewritten_query, reason}
           ├─ 低置信度/超时 → 降级方案一（规则路由）
           └─ 按 intent 路由:
               chat      → Ollama/低价模型
               knowledge → RAG
               tool      → MCP
               reasoning → 主模型
```

**分类 Prompt 设计**：
```
You are a query intent classifier. Categories:
- "chat": greetings, small talk, simple factual Q&A
- "knowledge": references to documents, knowledge bases
- "tool": invoke external tools/APIs
- "reasoning": complex analysis, coding, math, logic

Rewrite colloquial queries to clear form.
Output ONLY JSON: {"intent":"...","confidence":0.0-1.0,
                    "rewritten_query":"...","reason":"..."}
```

| 维度 | 评价 |
|------|------|
| 延迟 | ★★★☆☆ Ollama ~200ms / OpenAI ~500ms |
| 成本 | ★★★★☆ Ollama 免费；OpenAI ~380 tokens/次 |
| 准确率 | ★★★★☆ ~85-90%（语义理解好） |
| 维护性 | ★★★★☆ 改 Prompt 即可，无需代码变更 |
| Query改写 | ★★★★★ 原生支持 |
| 混合意图 | ★★☆☆☆ 排他路由，单 query 单模型 |
| 冷启动 | ★★★★★ 零（Prompt 即规则） |

**优点**：
- 零训练成本，Prompt 热更新即可调整分类逻辑
- 语义理解远超关键词匹配
- Query 改写提升下游 RAG 检索精度

**缺点**：
- 每次路由增加 200-500ms 延迟
- 改写可能引入 LLM 幻觉（脑补不存在的上下文）
- 排他路由不支持混合意图（如"先 RAG 检索再主模型推理"）

**适用**：成长期产品，需要语义理解且暂无标注数据积累。

---

### 3.7 方案六：渐进式混合路由（推荐演进态）

**原理**：将方案②（Embedding）+ 方案⑤（LLM分类）+ 方案①（规则兜底）叠成三层，按开销从低到高逐层决策。

```
                     ┌──────────────┐
                     │  User Query  │
                     └──────┬───────┘
                            │
               ┌────────────▼────────────┐
               │  L1: Embedding 快速匹配  │  0-5ms
               │  高置信度(>0.85) → 路由  │  免费
               └────────────┬────────────┘
                            │ 模糊 ~35%
               ┌────────────▼────────────┐
               │  L2: 轻量 LLM 分类      │  200ms
               │  中等置信度 → 路由       │  Ollama 免费
               └────────────┬────────────┘
                            │ 低置信度 ~10%
               ┌────────────▼────────────┐
               │  L3: 规则兜底           │  0ms
               │  + 主模型最终判断(可选)  │
               └─────────────────────────┘

整体: 65% query 走 L1(5ms), 25% 走 L2(200ms), 10% 走 L3(0ms)
加权延迟 ≈ 65%×5 + 25%×200 + 10%×0 ≈ 53ms
```

| 维度 | 评价 |
|------|------|
| 延迟 | ★★★★☆ 加权平均 ~53ms |
| 成本 | ★★★★★ 大部分 query 零附加成本 |
| 准确率 | ★★★★★ 多层互补 |
| 维护性 | ★★★☆☆ 需维护 Embedding 锚点 + Prompt |
| Query改写 | ★★★★☆ L2 支持（仅对需要改写的 query） |
| 冷启动 | ★★☆☆☆ 需准备 Embedding 锚点样例 |

**优点**：各方案取长补短——Embedding 负责快速分流，LLM 负责模糊 case，规则负责兜底。

**缺点**：系统复杂度上升，需要维护两套分类体系。

---

### 3.8 方案七：投机执行（Speculative Execution）

**原理**：并行调用 cheap 模型和主模型，cheap 先返回，若回答质量达标则 cancel 主模型调用。

```
0ms ┌─ Ollama.Generate ────────┐
    │                           ├─ 200ms: 完成 + 质检通过
    │                           └─ 返回 + cancel 主模型
    └─ MainModel.Generate ──────── [被 cancel]
```

| 维度 | 评价 |
|------|------|
| 延迟 | ★★★★☆ 取 cheap 模型的延迟 |
| 成本 | ★★★☆☆ 主模型 cancel 前已产生部分 token 费用 |
| 准确率 | ★★★★☆ 基于实际回答判断 |
| 维护性 | ★★★☆☆ |
| 流式兼容 | ★★☆☆☆ SSE cancel 语义不一致 |

**适用**：主模型按请求计费（非按 token）或本地部署可零成本 cancel 的特殊场景。

## 五、推荐路线

已实现方案⑥（三层渐进式混合路由）：

```
L1: Embedding 快速匹配 → 5ms → 拦截 ~65% 的高置信度 query
```

三层之间自动降级，加权平均延迟 ≈ 53ms。

**实现文件**：
- `common/aihelper/router_embedding.go` — L1 Embedding 快速意图匹配器
- `common/aihelper/router_llm.go` — L1+L2+L3 三层集成调度
- `common/aihelper/router.go` — 全局路由器单例 + 规则兜底（L3）
